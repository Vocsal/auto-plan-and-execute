# 各阶段产出文档模板

四个阶段的产出文档必须遵守以下结构。下游 LLM 和审查者依赖这些章节稳定地定位信息。

---

## 1. 计划方案文档 — `plan-v{N}.md` 或 `plan.md`（定稿）

```markdown
# 实现计划：{需求标题}

## 1. 背景与目标
{2-4 段说明业务背景、本次要解决的问题、成功的衡量标准}

## 2. 需求范围
### 2.1 包含
- {明确包含的功能/改动}
### 2.2 不包含
- {明确排除的内容，避免范围蔓延}

## 3. 现状分析
{基于代码探索，说明当前相关模块的实现、约束、技术债}

## 4. 方案设计
### 4.1 总体思路
{核心设计决策、关键技术选型、为什么这么做}
### 4.2 详细设计
{按模块/层次展开：数据模型、接口、流程、关键算法}
### 4.3 涉及变更的文件清单
- `path/to/file1.ext` — {变更说明}
- `path/to/file2.ext` — {新增/修改/删除}
{清单要尽量穷尽，是下游实施阶段的主要依据}

## 5. 实施步骤
按顺序列出可独立执行、可验证的步骤：
1. **步骤名** — 做什么 / 验证方式
2. ...

## 6. 风险与对策
| 风险 | 影响 | 对策 |
|---|---|---|
| ... | ... | ... |

## 7. 测试策略
- 单元测试：覆盖范围
- 集成/端到端测试：场景
- 手动验证：步骤

## 8. 回滚预案
{出问题时如何快速回退}
```

### 写作要求
- 完整、独立，下游不读其他文档就能照着实施
- **不要**写"v2 相比 v1 改进了..."、"采纳了审查报告中关于 X 的建议"
- 诚实标注未知点，放在"风险与对策"中
- 范围克制，不做无关重构

---

## 2. 计划审查报告 — `plan-review-v{N}.md`

```markdown
# 计划审查报告：{需求标题} — 第 {N} 轮

## 1. 审查范围
- 待审计划：`{plan 文档路径}`
- 审查时间：{今日日期}
- 审查维度：{列出实际覆盖的维度}

## 2. 总体评价
{2-3 段说明方案整体质量、最大亮点、最大隐患}

## 3. 问题清单

### P0（阻断）
- **[问题简称]**：详细描述
  - 位置：{方案章节/段落}
  - 影响：{说明后果}
  - 建议：{具体修改方向}

### P1（严重）
- ...

### P2（一般）
- ...

### P3（提示）
- ...

（无对应等级问题则写"无"）

## 4. 改进建议汇总
按优先级列出对计划撰写者的核心修改建议（3-7 条），便于下一轮迭代聚焦。

## 5. 通过/继续判定理由
- P0 数量：N
- P1 数量：N
- 判定：{解释为什么给出最终的 STATUS}

STATUS: PASS
```

`STATUS` 行参见 `references/status-protocol.md`。

---

## 3. 实施日志 — `execution-log-v{N}.md`

```markdown
# 实施日志：{需求标题} — 第 {N} 轮

## 1. 本轮目标
{首轮：完整落地计划方案。后续轮：解决上一轮审查报告中的 P0/P1 问题}

## 2. 计划符合度
| 计划中的步骤 | 状态 | 备注 |
|---|---|---|
| 步骤 1：xxx | 完成 / 部分完成 / 跳过 | {如有偏离说明原因} |
| 步骤 2：xxx | ... | ... |

## 3. 实际变更清单
- `path/to/file1.ext` — {新增/修改/删除}，{变更要点}
- ...

完整 diff 见 `git diff`。

## 4. 偏离计划的地方
{如果完全按计划落地，写"无"。否则列出每处偏离的原因和影响}

## 5. 测试与验证
- 单元测试：{命令、覆盖范围、结果}
- 集成测试：{...}
- Lint / Type Check：{结果}
- 手动验证：{步骤、结果}

## 6. 遗留问题
{已知但本轮未解决的问题，给下一轮或上线后跟进}

## 7. 上一轮审查报告问题处理（仅迭代时填）
| 审查问题 | 严重度 | 处理结果 | 处理方式 |
|---|---|---|---|
| ... | P0 | 已解决 / 部分解决 / 暂不处理 | {简述} |
```

---

## 4. 实施审查报告 — `execution-review-v{N}.md`

```markdown
# 实施审查报告：{需求标题} — 第 {N} 轮

## 1. 审查范围
- 计划方案：`{plan.md 路径}`
- 实施日志：`{execution-log 文档路径}`
- 审查时间：{今日日期}
- 检查的关键文件：{列出实际 Read 的文件}

## 2. 总体评价
{2-3 段说明实施整体质量、最大亮点、最大风险}

## 3. 计划符合度核查
| 计划步骤 | 实施状态 | 实际情况 | 是否符合 |
|---|---|---|---|
| 步骤 1：xxx | 完成 | ... | 是 / 否 / 部分 |

## 4. 问题清单

### P0（阻断）
- **[问题简称]**：详细描述
  - 位置：`file:line` 或描述
  - 影响：{说明后果}
  - 建议：{具体修复方向}

### P1（严重）
- ...

### P2（一般）
- ...

### P3（提示）
- ...

（无对应等级问题则写"无"）

## 5. 测试核查
- 是否复跑测试：是 / 否
- 复跑结果：{...}
- 测试覆盖评价：{...}

## 6. 改进建议汇总
按优先级列出对实施者的核心修复建议（3-7 条）。

## 7. 通过/继续判定理由
- P0 数量：N
- P1 数量：N
- 判定：{解释为什么给出最终的 STATUS}

STATUS: PASS
```

## 5. 最终交付总结 — `final-summary.md`

```markdown
# 项目需求实现总结：{需求标题}

## 1. 需求概述
{1-2 段说明本次解决的问题、目标}

## 2. 最终方案
{从定稿计划中提炼总体设计、关键决策、技术选型}

## 3. 实施成果
- 改动文件清单（基于实施日志整合）
- 新增/修改的接口或数据结构
- 测试覆盖情况

## 4. 验证结果
- 自动化测试结果
- 关键场景手动验证

## 5. 遗留问题与后续建议
{基于实施日志和最终审查报告整合}

## 6. 关键决策回顾
{基于多轮迭代过程中的重要权衡}

## 7. 回滚预案
{从计划回滚预案直接迁移并校对可行性}
```

### 写作要求
- 面向交付的文档，**不要**出现"v1/v2/审查指出"等迭代痕迹
- 信息完整自洽，读者无需翻看其他文档即可了解全貌
